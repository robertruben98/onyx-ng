# onyx-ng Plan C — DX, CI, Release

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`).

**Goal:** Make onyx-ng adoptable and releasable: consumer-facing docs, a CI quality gate, and a semantic-release publish-on-tag pipeline to private GitHub Packages. (Docs-site deploy to the VPS is done by invoking the existing `deploy-onyx` skill at a gated step — NOT built here.)

**Architecture:** Markdown docs + a Jest coverage gate; a GitHub Actions CI workflow mirroring the local commands; a semantic-release config that derives the version from conventional commits, injects it into the library source manifest, builds with ng-packagr, and publishes `dist/onyx-ui` to GitHub Packages on a version tag.

**Tech Stack:** GitHub Actions, semantic-release, Jest, Nx, ng-packagr (build already wired in Plan A).

## Global Constraints

- Branch `feature/distributable-v0.1` (NOT protected — a hook blocks them). No "Co-Authored-By"/"Generated with"/AI-attribution markers in commits (a hook blocks them).
- Package: `@robertruben98/onyx-ui`, private GitHub Packages `https://npm.pkg.github.com`. The publishable manifest is `libs/ui/components/package.json`; the built package is `dist/onyx-ui` (generated by `nx build ui-components`).
- Conventional commits are enforced by a git hook; the hook REJECTS the `!` breaking marker, so breaking changes are flagged via a `BREAKING CHANGE:` commit-body footer (semantic-release reads that).
- `postinstall: patch-package` applies the committed ng-packagr patch — CI's `npm ci` must run it (it does, via the lifecycle script).
- Do NOT trigger a real publish or a real VPS deploy in this plan — those are gated steps the controller runs after user confirmation. Config/workflows are authored and validated (dry-run / YAML parse) only.
- Existing local commands to mirror: `npm run build:tokens`, `npm run typecheck`, `npm run lint`, `npm test`, `npx nx build ui-components`, `npx nx build docs`. Keep the 224 tests green.

  push:
    branches: [main, agent-dialog-feature]

Note: `npm ci` triggers `postinstall: patch-package`; the ng-packagr build depends on it. Order matters: `build:tokens` before the library build (the lib copies `tokens.css`). Adjust node version if the repo pins a different one (check `.nvmrc`/`engines`).

### Task 3: semantic-release publish-on-tag

**Files:**

- Modify: root `package.json` (devDeps: semantic-release + plugins; a `release` script)

- Create: `.github/workflows/publish.yml`
- Create/append: `.npmrc` already exists (Plan A) — confirm it scopes `@robertruben98`.

**Interfaces:**

- Produces: a semantic-release config that, on a version tag / release run, derives the next version from conventional commits, sets it in `libs/ui/components/package.json`, builds, and publishes `dist/onyx-ui` to GitHub Packages; generates/updates `CHANGELOG.md` and a GitHub Release.

- [ ] **Step 1: Install semantic-release + plugins (dev)**

Run: `npm install -D semantic-release @semantic-release/commit-analyzer @semantic-release/release-notes-generator @semantic-release/changelog @semantic-release/exec @semantic-release/npm @semantic-release/git`

- [ ] **Step 2: Write `.releaserc.json`**

The version must reach the BUILT package (`dist/onyx-ui`) which ng-packagr generates from `libs/ui/components/package.json`. Use `@semantic-release/exec` to stamp the source manifest and build during `prepare`, then publish the built dir:

```json
{
  "branches": ["main"],
  "tagFormat": "v${version}",
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    ["@semantic-release/changelog", { "changelogFile": "CHANGELOG.md" }],
    [
      "@semantic-release/exec",
      {
        "prepareCmd": "npm version ${nextRelease.version} --no-git-tag-version --workspaces=false --prefix libs/ui/components && npm run build:tokens && npx nx build ui-components"
      }
    ],
    ["@semantic-release/npm", { "pkgRoot": "dist/onyx-ui" }],
    [
      "@semantic-release/git",
      {
        "assets": ["CHANGELOG.md", "libs/ui/components/package.json"],
        "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
      }
    ],
    "@semantic-release/github"
  ]
}
```

Note: `npm version --prefix libs/ui/components` bumps the source manifest; the subsequent `nx build ui-components` regenerates `dist/onyx-ui/package.json` with that version; `@semantic-release/npm` with `pkgRoot: dist/onyx-ui` publishes it. If `npm version` flags differ on the installed npm, adapt to a form that writes the version into `libs/ui/components/package.json` without tagging — and report it. The dist `package.json` already has `publishConfig.registry: https://npm.pkg.github.com`.

- [ ] **Step 3: Write `.github/workflows/publish.yml`**

```yaml
name: Release
on:
  push:
    tags: ["v*"]
  workflow_dispatch:

permissions:
  contents: write
  packages: write
  issues: write
  pull-requests: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
          registry-url: "https://npm.pkg.github.com"
      - run: npm ci
      - run: npm run build:tokens
      - run: npm test -- --ci
      - run: npx semantic-release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

`GITHUB_TOKEN` (the built-in Actions token, with the `packages: write` permission above) authenticates both the GitHub Release and the GitHub Packages publish.

- [ ] **Step 4: Add a `release` script + validate config loads**

Add `"release": "semantic-release"` to root `package.json` scripts. Validate the config parses and the plugin chain loads without publishing:
Run: `npx semantic-release --dry-run --no-ci 2>&1 | tail -30` (it will analyze commits and compute the next version; it may warn about auth/branch on a non-CI/non-main branch — that's expected. The goal is: no config/plugin-load errors, and it identifies the conventional commits incl. the `BREAKING CHANGE:` from the rename). Report the dry-run output summary (the computed next version + any errors). If dry-run can't run far without a token, at least confirm `.releaserc.json` is valid JSON and all referenced plugins are installed (`node -e "require('./.releaserc.json'); ['@semantic-release/exec','@semantic-release/npm','@semantic-release/git','@semantic-release/changelog'].forEach(p=>require.resolve(p))" && echo "config+plugins ok"`).

- [ ] **Step 5: Commit**

```bash
git add package.json package-lock.json .releaserc.json .github/workflows/publish.yml
git commit -m "ci: semantic-release publish-on-tag to GitHub Packages"
```
